# Chapter 2 — First Principles

Instead of proving identity with a password...

What if mathematics could prove identity?

That is exactly what SSH does.

---

# Public Key Cryptography

SSH creates TWO keys.

```
Private Key

Public Key
```

Notice something important.

It does **NOT** create

```
Password
```

Instead it creates

```
Key Pair
```

---

# Why Two Keys?

Imagine your house.

```
House Lock

House Key
```

Anyone can see the lock.

Only you own the key.

Exactly the same idea.

```
Public Key

↓

Lock
```

```
Private Key

↓

Key
```

The public key is useless without the matching private key.

---

# Private Key

Lives only on your computer.

```
MacBook

↓

~/.ssh/id_ed25519
```

Never

- Upload
- Email
- Commit
- Share

If someone gets this file...

They become you.

---

# Public Key

```
~/.ssh/id_ed25519.pub
```

Safe to share.

GitHub stores this key.

It contains NO secret.

Think of it like a lock anyone can inspect.

---

# Chapter 3 — Authentication

Suppose GitHub already has your public key.

```
GitHub

↓

Public Key
```

Your laptop has

```
Private Key
```

Now GitHub asks

> Prove you own the matching private key.

It sends a random challenge.

```
GitHub

↓

Random Challenge
```

Your laptop signs it.

```
Challenge

↓

Private Key

↓

Digital Signature
```

GitHub verifies the signature using

```
Public Key
```

If verification succeeds

```
✓ Access Granted
```

Notice something beautiful.

The private key

**never leaves your computer.**

This is why SSH is secure.

---

# Digital Signature

Think of signing a document.

```
Document

↓

Your Signature
```

Anyone who knows your signature can verify it.

SSH works similarly.

Except the signature is generated mathematically.

Every connection produces a new signature.

No passwords are transmitted.

# Chapter 4 — Generating SSH Keys

Modern SSH recommends

```
Ed25519
```

Instead of RSA.

Why?

| RSA | Ed25519 |
|------|----------|
| Older | Modern |
| Larger Keys | Smaller Keys |
| Slower | Faster |
| More Complex | Simpler |
| Excellent | Excellent + Recommended |

Generate a key

```bash
ssh-keygen -t ed25519 -C "your-email@example.com"
```

Explanation

```
ssh-keygen

↓

Generate SSH Keys
```

```
-t

↓

Type
```

```
ed25519

↓

Algorithm
```

```
-C

↓

Comment
```

The comment is only a label.

GitHub ignores it.

---

# Generated Files

```
~/.ssh/

id_ed25519

id_ed25519.pub
```

```
id_ed25519

↓

Private Key
```

```
id_ed25519.pub

↓

Public Key
```

# Chapter 5 — The SSH Agent

Imagine Git wants your key.

Without SSH Agent

```
Git

↓

Find Private Key

↓

Authenticate

↓

Forget
```

Every application must find the key.

Now imagine another process.

```
SSH Agent
```

```
Git

↓

SSH Agent

↓

Private Key
```

SSH Agent is simply

> A background process that securely stores unlocked private keys in memory.

Applications no longer search for keys.

They ask the SSH Agent.

---

# Starting the Agent

```bash
eval "$(ssh-agent -s)"
```

This starts

```
ssh-agent
```

as a background process.

---

# Adding a Key

```bash
ssh-add ~/.ssh/id_ed25519
```

Now the agent remembers the key.

Future authentication becomes instant.

# Chapter 6 — macOS Keychain

People often confuse

```
Private Key
```

with

```
Passphrase
```

These are different.

Imagine your private key itself is encrypted.

```
Private Key

↓

Encrypted

↓

Requires Passphrase
```

Without Keychain

```
Every reboot

↓

Enter Passphrase
```

With Keychain

```
SSH

↓

Keychain

↓

Passphrase

↓

Unlock Private Key
```

macOS remembers the passphrase securely.

Not the key.

The key remains inside

```
~/.ssh/
```

---

# Add to Keychain

```bash
ssh-add --apple-use-keychain ~/.ssh/id_ed25519
```

# Chapter 7 — SSH Configuration

SSH reads

```
~/.ssh/config
```

Think of this file as

> Routing Rules.

Exactly like

- Nginx configuration
- Git configuration
- Kubernetes YAML

It contains instructions.

Not secrets.

---

Example

```text
Host github.com
    AddKeysToAgent yes
    UseKeychain yes
    IdentityFile ~/.ssh/id_ed25519
```

---

## Host

```
Host github.com
```

Means

Whenever connecting to GitHub...

Use the following configuration.

---

## IdentityFile

```
IdentityFile ~/.ssh/id_ed25519
```

Without this

SSH tries every available key.

```
Key 1

↓

Rejected

↓

Key 2

↓

Rejected

↓

Key 3
```

With IdentityFile

```
GitHub

↓

Immediately use

↓

id_ed25519
```

Cleaner.

Faster.

---

## AddKeysToAgent

```
AddKeysToAgent yes
```

If SSH successfully uses this key

Automatically add it to the SSH Agent.

Without it

```
ssh-add
```

must often be run manually.

---

## UseKeychain

```
UseKeychain yes
```

macOS-specific.

Store passphrases inside Apple's Keychain.

Not the private key.

# Chapter 8 — Multiple Hosts

One configuration file can manage many identities.

Example

```text
Host github.com
    IdentityFile ~/.ssh/github_personal

Host github-work
    HostName github.com
    User git
    IdentityFile ~/.ssh/github_company

Host aws-prod
    HostName 54.xx.xx.xx
    User ubuntu
    IdentityFile ~/.ssh/aws_prod

Host digitalocean
    HostName 159.xx.xx.xx
    User root
    IdentityFile ~/.ssh/do_key
```

Now connecting becomes

```bash
ssh aws-prod
```

instead of

```bash
ssh -i ~/.ssh/aws_prod ubuntu@54.xx.xx.xx
```

Much simpler.

---

# Why Companies Use Multiple Keys

Imagine a senior backend engineer.

```
~/.ssh/

github_personal

github_company

aws_dev

aws_prod

gcp_cluster

azure_vm
```

Each key grants access to different systems.

If one key leaks

Only that key is revoked.

Everything else continues working.

This follows the Principle of Least Privilege.

# Chapter 10 — Complete Architecture

```
Git Push
    │
    ▼
Git
    │
    ▼
SSH
    │
    ▼
Read ~/.ssh/config
    │
    ├── Which Host?
    ├── Which Key?
    ├── Add To Agent?
    └── Use Keychain?
    │
    ▼
SSH Agent
    │
    ▼
Private Key
    │
    ▼
Generate Digital Signature
    │
    ▼
GitHub
    │
    ▼
Verify Signature
(using Public Key)
    │
    ▼
Authentication Success
```

Notice what never happens.

```
Private Key

❌ Never leaves your computer.
```

That single fact is why SSH authentication is one of the safest and most elegant authentication systems ever designed.

# Key Takeaways

- SSH solves authentication using mathematics instead of passwords.
- SSH generates a Public Key and a Private Key.
- The Private Key never leaves your machine.
- GitHub stores only your Public Key.
- Authentication happens using Digital Signatures.
- SSH Agent stores unlocked keys in memory.
- macOS Keychain stores passphrases, not private keys.
- `~/.ssh/config` tells SSH which key to use for each host.
- Multiple identities can coexist in a single configuration file.
- Understanding SSH is far more valuable than memorizing commands because the same principles apply to servers, cloud infrastructure, CI/CD pipelines, Kubernetes clusters, and production systems.
