# `pub` (Visibility)

## First Principle

By default, **everything in Rust is private**.

Think of your house.

```text
House

Bedroom
```

Can a stranger walk into your bedroom?

No.

Only you can access it.

Rust follows the same philosophy.

Unless you explicitly allow it, other modules **cannot** access your functions, structs, or fields.

---

# Private by Default

Suppose you have

```rust
// math.rs

fn add(a: i32, b: i32) -> i32 {
    a + b
}
```

And in `main.rs`

```rust
mod math;

fn main() {
    let answer = math::add(5, 10);
}
```

Rust gives an error.

```text
error:
function `add` is private
```

Why?

Because `add()` belongs only to the `math` module.

---

# Making it Public

Simply add `pub`.

```rust
// math.rs

pub fn add(a: i32, b: i32) -> i32 {
    a + b
}
```

Now

```rust
mod math;

fn main() {
    let answer = math::add(5, 10);

    println!("{}", answer);
}
```

works perfectly.

# `pub` Means

> "I allow other modules to use this."

---

# `pub` with Functions

Private

```rust
fn login() {}
```

Public

```rust
pub fn login() {}
```

---

# `pub` with Structs

Private

```rust
struct User {
    name: String,
}
```

Only this module knows `User` exists.

---

Public

```rust
pub struct User {
    name: String,
}
```

Now other modules can create a `User`.

---

# But Something Interesting Happens

Suppose

```rust
pub struct User {
    name: String,
}
```

Can another module do

```rust
let user = User {
    name: String::from("Steve"),
};
```

**No!**

Rust still gives an error.

Why?

Because the **field** is still private.

---

You must also write

```rust
pub struct User {
    pub name: String,
}
```

Now everything works.

---

# Think of It Like a Locker

The locker is public.

```text
Locker
```

But the items inside are locked.

You need another key.

```text
Locker

↓

Public

↓

Items

↓

Still Private
```

Rust lets you control both separately.

---

# Example

```rust
pub struct Student {
    pub name: String,
    pub age: u32,
}
```

Now another module can do

```rust
let student = Student {
    name: String::from("Leonard"),
    age: 24,
};
```

---

# Why Not Make Everything `pub`?

Imagine every function in Koel were public.

```text
JWT

Database

Password Hash

Internal Cache

Redis

OAuth

Email

Debug Functions
```

Any module could call anything.

That creates tight coupling and makes maintenance harder.

Instead, expose only what other modules actually need.

Everything else stays private.

---

# Real Koel Example

Imagine

```text
auth

↓

login.rs
```

Inside

```rust
pub fn login() {}
```

Other modules can call

```rust
auth::login();
```

But suppose

```rust
fn verify_password() {}
```

Only `login.rs` needs it.

Keep it private.

```rust
fn verify_password() {}
```

This hides implementation details.

---

# Mental Model

```text
Module

├── pub login()

├── pub signup()

├── verify_password()

├── hash_secret()
```

Other modules see

```text
login()

signup()
```

They **cannot** see

```text
verify_password()

hash_secret()
```

Those remain internal.

---

# Rule to Remember

- No `pub` → Private.
- `pub` → Visible outside the module.
- Structs and their fields have separate visibility.
- Only expose what other modules need.

# Solutions

## Solution 1

It **will not compile**.

Reason:

```rust
add()
```

is private because it is missing `pub`.

---

## Solution 2

```rust
// math.rs

pub fn add(a: i32, b: i32) -> i32 {
    a + b
}
```

Now `main.rs` can call `math::add()`.

---

## Solution 3

No.

Even though the `User` struct is public, the `name` field is still private.

---

## Solution 4

```rust
pub struct User {
    pub name: String,
}
```

Now other modules can access `user.name`.

---

## Solution 5

Public items are:

```rust
pub struct Book

pub title

pub fn create_book()
```

Private items are:

```rust
pages

calculate_pages()
```

---

## Solution 6

Rust follows the **principle of least privilege**.

By keeping items private by default, it:

- Prevents accidental misuse.
- Protects internal implementation details.
- Encourages clear module boundaries.
- Makes large projects easier to maintain.

---

## Solution 7

A sensible design would be:

```rust
pub fn login() {}

pub fn logout() {}

fn verify_password() {}

fn hash_password() {}
```

`login()` and `logout()` form the public API of the module.

`verify_password()` and `hash_password()` are implementation details used internally, so they should remain private.
