When beginners first see Rust code, they usually ask:

> **"Why is there a `?` everywhere?"**

By the end of this topic, you'll understand exactly why.

---

# The `?` Operator

## First Principle

Suppose you're doing three tasks in order:

1. Connect to the database.
2. Find the user.
3. Create a session.

Each step can fail.

```text
Connect Database

↓

Find User

↓

Create Session
```

If any step fails,

**stop immediately** and return the error.

That's exactly what `?` does.

---

# Without `?`

Imagine this function.

```rust
fn login() -> Result<User, String> {
    let user = match find_user() {
        Ok(user) => user,

        Err(error) => return Err(error),
    };

    Ok(user)
}
```

Let's read it slowly.

If

```text
find_user()

↓

Ok(User)
```

Continue.

But if

```text
find_user()

↓

Err(Error)
```

Immediately return

```rust
Err(error)
```

---

# This Pattern Appears Everywhere

Every database query.

Every file read.

Every network request.

Every HTTP request.

So Rust introduced a shortcut.

---

# With `?`

The same code becomes

```rust
fn login() -> Result<User, String> {
    let user = find_user()?;

    Ok(user)
}
```

Much shorter.

---

# What Does `?` Actually Do?

Think of it like this.

```text
Result

↓

Is it Ok?

↓

Yes

↓

Continue
```

or

```text
Result

↓

Is it Err?

↓

Yes

↓

Return immediately
```

---

## Visualization

```text
find_user()

↓

Result

├── Ok(User)

│

│ Continue

│

└── Err(Error)

↓

return Err(Error)
```

---

# `?` is Just Syntax Sugar

This

```rust
let user = find_user()?;
```

is almost exactly the same as

```rust
let user = match find_user() {
    Ok(value) => value,

    Err(error) => return Err(error),
};
```

The compiler expands it for you.

---

# Example

Suppose

```rust
fn get_age(user_exists: bool) -> Result<u32, String> {
    if user_exists {
        Ok(25)
    } else {
        Err("User not found".to_string())
    }
}

fn main() {
    match get_age(true) {
        Ok(age) => println!("Age: {}", age),
        Err(error) => println!("Error: {}", error),
    }

    match get_age(false) {
        Ok(age) => println!("Age: {}", age),
        Err(error) => println!("Error: {}", error),
    }
}
```

Output

```rust
Age: 25
Error: User not found
```

## Same example using ?

```rs
fn get_age(user_exists: bool) -> Result<u32, String> {
    if user_exists {
        Ok(25)
    } else {
        Err("User not found".to_string())
    }
}

fn main()  {
    let age = get_age(true)?;
    println!("Age: {}", age);

    let age = get_age(false)?; // Returns Err immediately
    println!("Age: {}", age);  // Never runs

    Ok(())
}
```

```text
Age: 25
Success
Error: User not found
```

## same example

```rs
fn get_age(user_exists: bool) -> Result<u32, String> {
    if user_exists {
        Ok(25)
    } else {
        Err("User not found".to_string())
    }
}

fn main_logic(user_exists: bool) -> Result<u32, String> {
    let age = get_age(user_exists)?; // If Err, return immediately

    println!("Age: {}", age);

    Ok(age)
}

fn main() {
    match main_logic(true) {
        Ok(_) => println!("Success"),
        Err(error) => println!("Error: {}", error),
    }

    match main_logic(false) {
        Ok(_) => println!("Success"),
        Err(error) => println!("Error: {}", error),
    }
}
```

When you write:

```rs
let age = get_age(user_exists)?;
```

Rust treats it like:

```rs
let age = match get_age(user_exists) {
Ok(value) => value,
Err(error) => return Err(error),
};
```

# Multiple `?`

Imagine

```rust
let user = find_user()?;

let token = create_token()?;

save_session()?;
```

Flow

```text
Find User

↓

Create Token

↓

Save Session

↓

Done
```

If **any** step fails

Everything stops immediately.

---

# Why This is Beautiful

Without `?`

```rust
match ...

match ...

match ...

match ...

match ...
```

Huge nested code.

With `?`

```rust
let user = find_user()?;

let token = create_token()?;

save_session()?;
```

Much cleaner.

---

# Important Rule

`?` only works on

```rust
Result
```

and

```rust
Option
```

Because both represent success/failure states.

---

# Another Important Rule

The function itself must return

```rust
Result<...>
```

or

```rust
Option<...>
```

Otherwise Rust doesn't know where to return the error.

---

# Mental Model

```text
Function

↓

find_user()?

↓

Ok

↓

Continue

────────────

Err

↓

Return Immediately
```

---

# Rule to Remember

- `?` only works with `Result` or `Option`.
- If the value is successful (`Ok`/`Some`), unwrap it and continue.
- If it's a failure (`Err`/`None`), return immediately.
- `?` replaces repetitive `match` statements.
- The surrounding function must also return `Result` or `Option`.

# Solutions

## Solution 1

```rust
let user = find_user()?;
```

The `?` operator automatically:

- Extracts the value if it's `Ok`.
- Returns the error immediately if it's `Err`.

---

## Solution 2

Execution stops immediately.

The surrounding function returns:

```rust
Err("JWT Failed")
```

Nothing after that line executes.

---

## Solution 3

The `Token` value is extracted.

Execution continues with the next line.

---

## Solution 4

No.

`i32` is just a number.

It doesn't represent success or failure.

The `?` operator only works with types that implement the `Try` behavior, such as `Result` and `Option`.

---

## Solution 5

No.

This will not compile because `main()` returns `()` (nothing), not a `Result` or `Option`.

For `?` to work, the surrounding function must also return a compatible type, for example:

```rust
fn main() -> Result<(), String> {
    let user = find_user()?;
    Ok(())
}
```

---

## Solution 6

Execution flow:

```text
find_user()

↓

Success

↓

create_session()

↓

Error

↓

Return Immediately
```

`send_email()` is **never executed** because `?` returns as soon as it encounters the error.

---

## Solution 7

It is roughly equivalent to:

```rust
let age = match get_age() {
    Ok(value) => value,
    Err(error) => return Err(error),
};
```

The `?` operator is simply a concise way of writing this common pattern.

# The Big Picture

We've now built a complete chain:

```text
Option<T>

↓

Some / None
```

```text
Result<T, E>

↓

Ok / Err
```

```text
?

↓

Automatically handle
Ok / Err
```

These three concepts work together constantly in real Rust code.

For example, a SQLx query often looks like this:

```rust
let user = sqlx::query_as!(User, "SELECT * FROM users WHERE id = $1", id)
    .fetch_one(&pool)
    .await?;
```

You can now read it as:

1. Run the query.
2. Wait for it to finish (`.await`).
3. If it succeeds, give me the `User`.
4. If it fails, return the error immediately.

That's exactly what experienced Rust developers see when they read this code.
