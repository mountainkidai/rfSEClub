# 1.3 Stack vs Heap

This is one of the most important concepts in programming.

If you deeply understand this:

- React state becomes obvious
- performance becomes obvious
- memory leaks become obvious
- closures become obvious

Most developers memorize syntax.
Elite engineers understand memory.

---

# First Principle

A computer has RAM.

Your program needs places to store data.

JavaScript engines split memory into two major areas:

# 1. Stack

and

# 2. Heap

That’s it.

---

# Think Like an Operating System

Imagine a huge warehouse.

You need two storage systems:

---

## Stack = Fast Small Shelf

Used for:

- tiny data
- temporary data
- function calls
- execution tracking

VERY FAST.

But limited.

---

## Heap = Giant Storage Room

Used for:

- large data
- objects
- arrays
- functions
- complex structures

Flexible.

But slower.

---

# Stack (Simple Mental Model)

The stack works like plates.

Last in → first out.

(LIFO)

Example:

```js
function greet() {
  const name = "Pavel";
}

greet();
```

When `greet()` runs:

Stack gets:

```txt
greet()
name = "Pavel"
```

After function finishes:

Everything is removed instantly.

Very efficient.

---

# Why Stack Is Fast

Because memory is sequential.

Engine always knows:

> where the next item goes

No searching.

No fragmentation.

Just push/pop.

That’s why function calls are insanely fast.

---

# Heap (Simple Mental Model)

Now objects.

```js
const user = {
  name: "Pavel",
  age: 40,
};
```

Objects are too dynamic for stack.

Why?

Because:

- size can change
- properties can change
- nested objects exist

So engine stores object in heap.

---

# Important

The actual object lives in heap.

But the variable lives in stack.

```txt
Stack:
user → reference/address

Heap:
{ name: "Pavel", age: 40 }
```

This changes EVERYTHING.

---

# Primitive vs Reference (Now It Makes Sense)

You already finished 1.2.

Now connect it.

---

## Primitive

```js
let a = 5;
let b = a;
```

Copy actual value.

```txt
Stack:
a = 5
b = 5
```

Independent.

---

## Object

```js
const a = { score: 10 };
const b = a;
```

Now:

```txt
Stack:
a → 0x123
b → 0x123

Heap:
0x123 → { score: 10 }
```

Both point to SAME object.

So:

```js
b.score = 99;
```

Changes shared heap object.

---

# Why React Cares About This

React compares references.

Not deep objects.

```js
setUser(user);
```

If reference same:

React may skip render.

But:

```js
setUser({ ...user });
```

Creates NEW heap object.

New reference.

React re-renders.

This single concept is worth millions.

---

# Closures Also Depend On Heap

Example:

```js
function counter() {
  let count = 0;

  return function () {
    count++;
    console.log(count);
  };
}
```

Normally local variables die after function ends.

But closure keeps reference alive.

So engine stores needed data in heap.

This is why closures persist memory.

---

# A closure keeps outer variables alive as long as some function still references them.

Example:

```js
function counter() {
  let count = 0;

  return function () {
    count++;
    console.log(count);
  };
}

const c = counter();
```

Here:

```text
counter() finished running
but the returned function still uses count
so JavaScript keeps count alive

As long as c exists, count stays in memory.
```

# Stack Overflow

Infinite recursion:

```js
function test() {
  test();
}
```

Every call adds new stack frame.

Eventually stack fills.

Crash:

```txt
Maximum call stack size exceeded
```

Because stack is LIMITED.

# Heap Problems — From First Principles

Your app is a machine.

Memory is fuel.

The heap is the warehouse storing data:

- objects
- arrays
- images
- closures
- DOM elements

Example:

```js id="t8wo58"
const users = [{ name: "Ali" }];
```

That data goes into the heap.

---

# Core Law

More objects = more memory.

If old objects are not removed:

- memory grows
- cleanup becomes expensive
- app slows

---

# Garbage Collector (GC)

JavaScript has an automatic cleaner.

It periodically asks:

> “What data is no longer needed?”

Unused data gets deleted.

But there’s a cost:

- bigger heap
- longer scanning
- more pauses

---

# Memory Leak

A memory leak is:

> Data that should die but stays alive.

Example:

```js id="p0sx8x"
let cache = [];

setInterval(() => {
  cache.push(new Array(100000));
}, 100);
```

Heap grows forever.

The app becomes heavier every second.

---

# Why Apps Freeze

GC must inspect memory.

Small heap:

- fast cleanup

Huge heap:

- expensive cleanup
- UI pauses

User feels:

- laggy scrolling
- delayed clicks
- frozen animations

---

# Why Mobile Crashes

Phones have limited RAM.

A bloated heap can:

- kill browser tabs
- reload apps
- drain battery

Desktop hides bad engineering.
Mobile exposes it.

---

Fast apps:

- reduce bounce
- increase engagement

Slow apps:

- lose money

Milliseconds compound at scale.

Heap = long-term memory storage.

Too much memory causes:

1. slow cleanup
2. UI freezes
3. battery drain
4. crashes

Frontend performance is largely:

> controlling memory growth.

Fast + small.
