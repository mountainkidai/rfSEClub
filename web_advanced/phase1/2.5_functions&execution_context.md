# 2.5 Function Declarations vs Expressions

Most developers think this is syntax trivia.

It’s not.

This topic teaches:

- hoisting
- execution context
- memory creation
- JavaScript parsing behavior

This is runtime architecture.

---

# First Principle

In JavaScript:

> functions are values

You can:

- store them
- pass them
- return them
- assign them

This is why JavaScript became powerful.

---

# Two Main Ways To Create Functions

## 1. Function Declaration

```js id="fvqjlwm"
function greet() {
  console.log("hello");
}
```

---

## 2. Function Expression

```js id="6mlj2o"
const greet = function () {
  console.log("hello");
};
```

Looks similar.

Completely different behavior internally.

---

# The Real Difference

The difference is:

# WHEN JavaScript creates them.

Not how they look.

---

# JavaScript Runs In 2 Phases

Whenever file/function executes:

---

## Phase 1 — Creation Phase

Engine scans code.

Allocates memory.

Sets up execution context.

---

## Phase 2 — Execution Phase

Runs line by line.

---

# Function Declarations Are Fully Hoisted

Example:

```js id="clspfd"
sayHello();

function sayHello() {
  console.log("hello");
}
```

Works.

Why?

During creation phase:

```txt id="skkhjf"
Memory:
sayHello → full function
```

Function already exists before execution starts.

---

# Mental Model

Function declarations are loaded into memory entirely before code runs.

Almost like:

```txt id="cfzv4t"
Pre-installed into execution context
```

---

# Function Expressions Behave Differently

Example:

```js id="zjicge"
sayHello();

const sayHello = function () {
  console.log("hello");
};
```

Crash.

```txt id="smmby9"
Cannot access before initialization
```

Why?

Because only variable is hoisted.

NOT function value.

---

# What Actually Happens

Creation phase:

```txt id="7j8w1m"
sayHello → uninitialized
```

Execution phase later assigns:

```js id="4ml0qt"
sayHello = function () {};
```

Before assignment:

function does not exist yet.

---

# Important Distinction

## Function Declaration

```js id="q98gbk"
function test() {}
```

Engine stores:

```txt id="5mqe93"
test → actual function
```

Immediately.

---

## Function Expression

```js id="7u07hn"
const test = function () {};
```

Engine stores:

```txt id="d6wsn7"
test → uninitialized
```

Then later:

```txt id="3ij2xw"
test → function
```

---

# Why `const` Matters

Imagine:

```js id="6hr8zh"
var test = function () {};
```

Then:

```js id="8v97s2"
console.log(test);
```

Before assignment:

```txt id="zkd3cv"
undefined
```

Because `var` initializes as undefined.

But:

```js id="6w6mwa"
const test = function () {};
```

Exists in Temporal Dead Zone.

Safer.

---

# Named vs Anonymous Expressions

Expression can also have name.

```js id="01z9fq"
const greet = function hello() {
  console.log("hi");
};
```

Outer variable:

```txt id="0njylh"
greet
```

Inner function name:

```txt id="slazkr"
hello
```

Useful for recursion/debugging.

---

# Arrow Functions Are Expressions

```js id="lshsje"
const add = (a, b) => a + b;
```

This is still:

# function expression

Not declaration.

Meaning:

- not fully hoisted
- behaves like variable assignment

---

# Why React Uses Expressions Everywhere

Example:

```js id="24ndzt"
const Button = () => {};
```

Why not declarations?

Because expressions:

- fit modular architecture
- avoid accidental hoisting issues
- align with functional patterns
- behave predictably with imports

Modern JS ecosystems favor expressions.

---

# Real Runtime Insight

Declarations are processed EARLY.

Expressions are processed DURING execution.

This changes:

- initialization order
- dependency behavior
- circular imports
- module loading

In large systems, this matters massively.

---

# Common Interview Trap

```js id="5qom9m"
console.log(foo);

function foo() {}
```

Output:

```txt id="1r8hcf"
[Function: foo]
```

But:

```js id="r1qz0n"
console.log(foo);

var foo = function () {};
```

Output:

```txt id="5lqps5"
undefined
```

Because variable hoisted, assignment not.

---

# Billion Dollar Insight

Large systems fail from:

- initialization bugs
- hidden dependencies
- unpredictable execution order

Elite engineers understand:

> when memory gets allocated

Not just syntax.

This affects:

- bundlers
- SSR
- React hydration
- module systems
- performance optimization

Understanding execution timing is architecture-level thinking.

---

# Final Compression

## Function Declaration

```js id="n9qv56"
function test() {}
```

- fully hoisted
- available before execution
- loaded during creation phase

---

## Function Expression

```js id="0w61n2"
const test = function () {};
```

- variable hoisted
- function assigned later
- created during execution

---

# Golden Rule

Declarations exist before code runs.

Expressions exist only after assignment.

That single rule explains almost all behavior differences.
