# 1.4 Garbage Collection Basics

Most developers think:

> “Memory disappears automatically.”

Wrong.

Memory is one of the biggest hidden costs in software.

If memory keeps growing:

- app becomes slow
- mobile phones crash
- tabs freeze
- batteries drain
- servers become expensive

Garbage Collection (GC) exists to solve this.

---

# First Principle

When your program creates data:

```js
const user = {
  name: "Pavel",
};
```

Memory is allocated.

Usually in heap.

But later:

- maybe you stop using it
- maybe component unmounts
- maybe request finishes

Then:

> who removes unused memory?

That’s Garbage Collection.

---

# Simple Definition

Garbage Collector =

A system that finds unused memory and frees it.

---

# Think Like a City

Imagine heap memory as a city.

Objects are buildings.

Some buildings are still connected:

- roads
- electricity
- water

Those are ACTIVE objects.

Others are abandoned.

No connections.

No access.

Those are garbage.

GC demolishes abandoned buildings.

---

# Reachability (CORE CONCEPT)

JavaScript engines ask:

> “Can this object still be reached?”

If YES → keep it.

If NO → delete it.

This is the entire game.

---

# Example 1 — Reachable Object

```js
const user = {
  name: "Pavel",
};
```

`user` variable points to object.

```txt
Stack:
user → 0x123

Heap:
0x123 → { name: "Pavel" }
```

Object is reachable.

GC keeps it.

---

# Example 2 — Unreachable Object

```js
let user = {
  name: "Pavel",
};

user = null;
```

Now:

```txt
Stack:
user = null

Heap:
0x123 → { name: "Pavel" }
```

Nothing points to object anymore.

No references.

Unreachable.

GC eventually removes it.

---

# Important

GC is NOT instant.

Engine runs it periodically.

Why?

Because cleaning memory is expensive.

If GC ran constantly:

app would freeze.

---

# Mark and Sweep (Basic Algorithm)

Modern JS engines use variations of this.

Two phases:

---

## 1. Mark

Engine starts from ROOTS:

- global variables
- active functions
- current execution contexts
- closures

Then follows references.

Everything reachable gets marked.

---

## 2. Sweep

Anything NOT marked:

deleted from memory.

Simple.

---

# Closures Can Prevent GC

# Simple Closure Example

```js id="7l2q0o"
function outer() {
  let name = "Ali";

  return function () {
    console.log(name);
  };
}

const fn = outer();
```

---

# Step 1 — outer() runs

```text id="jlwm49"
STACK
┌────────────────┐
│ outer()        │
│ name = "Ali"   │
└────────────────┘
```

Normally:

- when function ends
- local variables disappear

So normally `"Ali"` should die.

---

# Step 2 — Inner function is returned

```js id="g6zq7q"
return function () {
  console.log(name);
};
```

This inner function uses `name`.

So JavaScript says:

> "I must keep `name` alive."

---

# Step 3 — Closure Created

```js id="3ybjlwm"
const fn = outer();
```

Now `outer()` finished.

BUT:

```text id="jlwm0g"
fn
 ↓
inner function
 ↓
remembers
 ↓
name = "Ali"
```

---

# Visual Memory Diagram

```text id="o4dr8q"
STACK
┌────────────────┐
│ fn = reference │
└────────────────┘


HEAP
┌─────────────────────┐
│ Closure Environment │
│ name = "Ali"        │
└─────────────────────┘
          ▲
          │
┌─────────────────────┐
│ inner function      │
└─────────────────────┘
```

---

# Step 4 — Calling fn()

```js id="ndrdpc"
fn();
```

Output:

```text id="7g1dzw"
Ali
```

Why?

Because the function remembered `name`.

---

# Key Idea

Closure means:

```text id="jlwmzd"
function
+
remembered variables
```

---

# Why GC Cannot Delete It

Garbage Collector asks:

```text id="5x2yq7"
"Can something still access name?"
```

YES:

```text id="pr25fx"
fn
 ↓
function
 ↓
name
```

So memory stays alive.

---

# Simple Memory Leak Example

```js id="t3dr0s"
function app() {
  let hugeData = new Array(1000000);

  return function () {
    console.log("clicked");
  };
}

const clickHandler = app();
```

Even though:

```js id="3djlwm"
hugeData;
```

is never used later,
the closure may still keep it alive.

---

# Final Mental Model

```text id="jlwm2w"
Normal function:
variables die after function ends

Closure:
variables survive because inner function remembers them
```

# Event Listeners Cause Leaks

# First Principles

A website is not “a page.”

It is a live machine.

The browser builds a giant tree in memory representing the UI.

That tree is the DOM.

DOM = Document Object Model.

---

# Simple DOM Example

HTML:

```html id="m2jlwm"
<button>Buy</button>
```

Browser turns it into an object in memory:

```text id="jlwm8a"
DOM
┌───────────────┐
│ button object │
│ text = "Buy"  │
└───────────────┘
```

The button on screen is actually a JavaScript object in memory.

---

# Event Listener

When user clicks something,
browser needs to know what to do.

So you attach a function:

```js id="7jlwmq"
button.addEventListener("click", () => {
  console.log("buy");
});
```

```js
function handleClick() {
  console.log("buy");
}

button.addEventListener("click", handleClick);
```

Now browser stores:

```text id="jlwm3q"
button
 ↓
click listener
 ↓
function
```

---

# Closures Enter

Example:

```js id="jlwm1f"
const userData = hugeData;

button.addEventListener("click", () => {
  console.log(userData);
});
```

The function remembers `userData`.

Now:

```text id="jlwm9u"
button
 ↓
listener
 ↓
function
 ↓
userData
```

Everything becomes connected.


# Why GC Cannot Clean It

GC asks:

```text id="4jlwmo"

```

YES.

Path still exists:

```text id="8jlwmc"

```

# First-Principles Rule

GC only removes unreachable memory.

As long as references exist:

- memory stays alive

Even if:

- element disappeared visually
- user cannot see it
- app no longer needs it

# Fix Event Listener Leaks

## Bad

```js id="2jlwm7"
const userData = hugeData;

button.addEventListener("click", function () {
  console.log(userData.name);
});

button.remove();
```

Problem:

```text id="3jlwm5"
listener
 ↓
userData
```

Memory may survive.

---

# Good Fix

Remove the listener too.

```js id="9jlwm5"
function handleClick() {
  console.log(userData.name);
}

button.addEventListener("click", handleClick);

// cleanup
button.removeEventListener("click", handleClick);

button.remove();
```

Now:

```text id="4jlwma"
button removed
listener removed
↓
no references
↓
GC cleans memory
```

---

# Why This Works

Before:

```text id="4jlwmj"
listener → userData
```

After removing listener:

```text id="7jlwms"
(no path exists)
```

GC sees:

> “Nothing can reach this data anymore.”

So memory is freed.

# Modern Frameworks

Frameworks like React help automate cleanup.

Example:

```js id="9jlwmg"
useEffect(() => {
  button.addEventListener("click", handleClick);

  return () => {
    button.removeEventListener("click", handleClick);
  };
}, []);
```

The returned function is cleanup logic.

# React Memory Leaks

Common causes:

- forgotten intervals
- subscriptions
- websockets
- event listeners
- stale closures

Example:

```js
useEffect(() => {
  const id = setInterval(fetchData, 1000);

  return () => clearInterval(id);
}, []);
```

Cleanup matters because GC only removes unreachable memory.

# Why GC Can Hurt Performance

GC pauses execution.

Large heap:

- longer cleanup
- UI freezes
- dropped frames
- mobile lag

This is why:

- unnecessary objects hurt
- excessive rerenders hurt
- large allocations hurt

Fast apps create LESS garbage.
