# Topic 1.1 — `var`, `let`, `const`

We will understand:

- memory behavior
- scope
- hoisting
- Temporal Dead Zone
- why `const` exists
- why React developers prefer `const`

from first principles.

---

# 🧠 First Principle

JavaScript executes code inside something called:

# Execution Context

Think of it like:

```txt id="s6jlwm"
a runtime container
```

Inside it, JavaScript stores:

- variables
- functions
- scope information

---

# Example

```js id="jlwm90"
let age = 25;
```

JavaScript creates memory for:

```txt id="jlwm91"
age → 25
```

Simple idea.

But how variables behave depends on:

- `var`
- `let`
- `const`

---

# 🔥 Before ES6

JavaScript originally only had:

```js id="jlwm92"
var
```

This caused many problems.

So ES6 introduced:

- `let`
- `const`

---

# PART 1 — `var`

Example:

```js id="jlwm93"
var name = "Aman";
```

---

# ⚡ Key Behavior 1

# Function Scoped

`var` ignores block scope.

Example:

```js id="jlwm94"
if (true) {
  var x = 10;
}

console.log(x);
```

Output:

```txt id="jlwm95"
10
```

WHY?

Because:

```txt id="jlwm96"
var only respects function boundaries
```

NOT `{}` blocks.

---

# 🚨 This Causes Bugs

Example:

```js id="jlwm97"
for (var i = 0; i < 3; i++) {}

console.log(i);
```

Output:

```txt id="jlwm98"
3
```

Variable leaked outside loop.

Bad design.

---

# ⚡ Key Behavior 2

# Hoisting

This is VERY important.

Example:

```js id="jlwm99"
console.log(a);

var a = 10;
```

Output:

```txt id="jlwm9a"
undefined
```

NOT error.

---

# 🧠 Why?

Because JavaScript internally does:

```js id="jlwm9b"
var a;

console.log(a);

a = 10;
```

Variable declaration moved to top.

This is:

# hoisting

---

# 🚨 Important

Only declaration is hoisted.

NOT assignment.

---

# Internal Memory View

Creation phase:

```txt id="jlwm9c"
a → undefined
```

Execution phase:

```txt id="jlwm9d"
a → 10
```

---

# 🚨 Problem With `var`

Problems:

- leaks scope
- hoisting confusion
- accidental overwrites
- unpredictable behavior

This is why modern JS rarely uses `var`.

---

# PART 2 — `let`

Introduced in ES6.

Example:

```js id="jlwm9e"
let age = 25;
```

---

# ⚡ Key Difference

# Block Scoped

Example:

```js id="jlwm9f"
if (true) {
  let x = 10;
}

console.log(x);
```

Output:

```txt id="’wini9g"
ReferenceError
```

Because:

```txt id="’wini9h"
let respects {}
```

Much safer.

---

# ⚡ Hoisting Still Exists

Many people misunderstand this.

`let` IS hoisted.

BUT differently.

---

# Example

```js id="’wini9i"
console.log(a);

let a = 10;
```

Output:

```txt id="’wini9j"
ReferenceError
```

NOT undefined.

---

# 🧠 Why?

Variable exists internally,
but is:

# uninitialized

Until execution reaches declaration.

---

# This Is Called

# Temporal Dead Zone (TDZ)

Time between:

```txt id="’wini9k"
variable creation
and
initialization
```

---

# Internal View

Creation phase:

```txt id="’wini9l"
a → uninitialized
```

Execution reaches:

```js id="’wini9m"
let a = 10;
```

Now:

```txt id="’wini9n"
a → 10
```

---

# 🚨 Why TDZ Is GOOD

It prevents accidental early access.

Safer code.
Fewer bugs.

---

# PART 3 — `const`

Example:

```js id="’wini9o"
const PI = 3.14;
```

---

# ⚡ Rules of `const`

---

# 1. Must initialize immediately

BAD:

```js id="’wini9p"
const x;
```

Error.

---

# 2. Cannot reassign

```js id="’wini9q"
const x = 10;

x = 20;
```

Error.

---

# 🚨 MOST IMPORTANT CONCEPT

`const` does NOT make objects immutable.

Example:

```js id="’wini9r"
const user = {
  name: "Aman",
};

user.name = "Rahul";
```

This WORKS.

---

# 🧠 Why?

Because:

```txt id="’wini9s"
const protects reference
NOT object contents
```

Think of a variable like a **label** stuck on a box in memory.

```js
const user = {
  name: "Aman",
};
```

Here:

- `user` is the label
- The object `{ name: "Aman" }` is the box stored somewhere in memory

JavaScript internally does something like:

```txt
user → memory address 0x001
```

At address `0x001`, the object lives:

```txt
0x001 = { name: "Aman" }
```

Now:

```js
user.name = "Rahul";
```

This changes the content INSIDE the box:

```txt
0x001 = { name: "Rahul" }
```

The label `user` is still pointing to the SAME address (`0x001`).

So `const` allows this.

---

## What `const` actually prevents

`const` prevents changing the variable to point somewhere else.

This is NOT allowed:

```js
user = {
  name: "Vikram",
};
```

Because now you are trying to make:

```txt
user → 0x002
```

instead of:

```txt
user → 0x001
```

And `const` says:

> "No. This label must always point to the same memory location."

---

# Easy Rule to Remember

## `const` protects:

✅ the reference (memory address)

## `const` does NOT protect:

❌ object properties
❌ array items
❌ nested values

---

## Same thing with arrays

```js
const arr = [1, 2, 3];

arr.push(4); // ✅ works
```

Because the array itself is modified, not the reference.

But:

```js
arr = [5, 6]; // ❌ error
```

Because now the reference changes.

---

# If you want true immutability

Use:

```js
Object.freeze(user);
```

Example:

```js
const user = Object.freeze({
  name: "Aman",
});

user.name = "Rahul"; // ❌ won't change
```

Now the object contents are protected too.

---

# 🚨 React Connection

This matters enormously in React.

React uses:

# reference comparison

Mutating objects causes:

- rerender bugs
- stale state
- memoization failures

---

# ⚡ Why React Devs Prefer `const`

Most variables should NOT be reassigned.

Example:

```js id="’wini9w"
const users = [];
```

Array contents may change,
but variable identity remains stable.

Safer mental model.

---

# 🌐 Scope Hierarchy

```txt id="’wini9x"
Global Scope
  ↓
Function Scope
  ↓
Block Scope
```

---

# `var`

Uses:

```txt id="’wini9y"
function scope
```

---

# `let` / `const`

Use:

```txt id="’wini9z"
block scope
```

---

# 🚨 Real Production Insight

Elite frontend apps avoid:

```js id="umillua0"
var
```

almost entirely.

Because predictable scope behavior matters in large systems.

---

# ⚡ Golden Rule

Use:

- `const` by default
- `let` when reassignment needed
- avoid `var`

---
