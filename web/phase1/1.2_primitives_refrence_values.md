# Topic 1.2 — Primitive vs Reference Values

This topic is MASSIVELY important.

Without this:

- React state feels random
- rerenders become confusing
- mutation bugs appear
- memoization breaks
- equality checks make no sense

This topic is foundational for:

- React
- Next.js
- Zustand
- Redux
- caching
- performance optimization

---

# 🧠 First Principle

In JavaScript,
values are stored in TWO fundamentally different ways:

# 1. Primitive Values

# 2. Reference Values

This distinction affects:

- memory
- copying
- comparison
- mutation
- performance

---

# PART 1 — Primitive Values

Primitives are simple immutable values.

Main primitives:

```txt id="jlwmb0"
string
number
boolean
null
undefined
symbol
bigint
```

Example:

```js id="jlwmb1"
const age = 25;
```

---

# 🧠 Mental Model

Primitive stores:

# actual value directly

Example:

```txt id="jlwmb2"
age → 25
```

Simple.

---

# Primitive Copying

Example:

```js id="jlwmb3"
let a = 10;
let b = a;

b = 20;
```

Now:

```txt id="jlwmb4"
a → 10
b → 20
```

Why?

Because primitives are:

# copied by value

`b` gets its OWN copy.

---

# 🚨 Primitive Values Are Immutable

Example:

```js id="jlwmb5"
let name = "Aman";

name[0] = "R";
```

Does NOT modify string.

Because strings are immutable.

Instead:

```js id="jlwmb6"
name = "Rahul";
```

creates NEW value.

---

# PART 2 — Reference Values

Objects are DIFFERENT.

Examples:

```js id="jlwmb7"
{}
[]
function(){}
new Map()
new Set()
```

These are:

# reference values

---

# Example

```js id="’winilb8"
const user = {
  name: "Aman",
};
```

---

# 🧠 Mental Model

Object itself lives somewhere in memory.

Variable stores:

# reference (address)

Example:

```txt id="’winilb9"
user → 0x001
```

At address `0x001`:

```txt id="’winilba"
{
  name: "Aman"
}
```

---

# 🚨 THIS CHANGES EVERYTHING

When copying object:

```js id="’winilbb"
const a = { x: 1 };

const b = a;
```

You are NOT copying object.

You are copying:

# reference

---

# Internal View

```txt id="’winilbc"
a → 0x001
b → 0x001
```

Both point to SAME object.

---

# So Mutation Affects Both

Example:

```js id="’winilbd"
b.x = 999;
```

Now:

```txt id="’winilbe"
a.x → 999
b.x → 999
```

Because SAME underlying object.

---

# 🚨 This Is One of the Biggest JS Confusions

People think:

```js id="’winilbf"
const b = a;
```

creates new object.

It does NOT.

It creates another reference.

---

# ⚡ Equality Difference

Primitives compare by:

# value

Objects compare by:

# reference

---

# Primitive Example

```js id="’winilbg"
10 === 10;
```

Result:

```txt id="’winilbh"
true
```

---

# Object Example

```js id="’winilbi"
{} === {}
```

Result:

```txt id="’winilbj"
false
```

WHY?

Because:

```txt id="’winilbk"
0x001 !== 0x002
```

Different references.

Even if contents identical.

---

# 🚨 The Real Reason React Uses Immutable Updates

React needs a FAST way to know:

> “Did state change?”

Instead of checking every property deeply,
React usually checks:

```js
oldState === newState;
```

This is called:

# reference equality

---

# First Principles

Objects live somewhere in memory.

Example:

```js
const user = {
  name: "Aman",
};
```

Memory:

```txt
user ──> 0x001
```

At address `0x001`:

```txt
{
  name: "Aman"
}
```

---

# React State Example

Suppose React state is:

```js
const [user, setUser] = useState({
  name: "Aman",
  age: 22,
});
```

---

# ❌ WRONG WAY (Mutation)

```js
user.name = "Rahul";

setUser(user);
```

Looks correct.

But here's the problem.

---

# What Actually Happened?

You changed contents INSIDE the same object.

Before:

```txt
0x001
{
  name: "Aman",
  age: 22
}
```

After:

```txt
0x001
{
  name: "Rahul",
  age: 22
}
```

Notice:

```txt
same memory address
```

Still:

```txt
user ──> 0x001
```

---

# What React Sees

Internally React compares:

```js
oldUser === newUser;
```

In this case:

```txt
0x001 === 0x001
```

Result:

```js
true;
```

React may think:

```txt
"Nothing changed"
```

Because reference is SAME.

---

# Why Is This Dangerous?

React optimizations depend on reference changes.

Mutation can cause:

- stale UI
- skipped rerenders
- broken memoization
- unpredictable bugs

---

# ✅ CORRECT WAY (Immutable Update)

Create a NEW object.

```js
setUser({
  ...user,
  name: "Rahul",
});
```

---

# What `...user` Means

```js
...user
```

copies old properties into a NEW object.

Equivalent to:

```js
setUser({
  name: user.name,
  age: user.age,
  name: "Rahul",
});
```

Final object becomes:

```js
{
  name: "Rahul",
  age: 22
}
```

---

# Important Part

This creates a BRAND NEW object in memory.

Old object:

```txt
0x001
{
  name: "Aman",
  age: 22
}
```

New object:

```txt
0x002
{
  name: "Rahul",
  age: 22
}
```

Now:

```txt
oldUser ──> 0x001
newUser ──> 0x002
```

Different references.

---

# What React Sees Now

```js
oldUser === newUser;
```

becomes:

```txt
0x001 === 0x002
```

Result:

```js
false;
```

Now React instantly knows:

```txt
"State changed → rerender"
```

---

# Why React Uses This Strategy

Because checking references is VERY fast.

```js
oldUser === newUser;
```

is instant.

But deeply checking every property:

```js
user.address.city.zip...
```

for huge objects would be expensive.

So React uses:

# immutable updates + reference comparison

for speed and scalability.

---

# Easy Mental Model

## Mutation

```txt
same box
changed contents
```

Harder to track.

---

## Immutable Update

```txt
new box
copied old contents
changed one value
```

Easy for React to detect.

---

# Final Rule

## Never mutate React state directly

❌ Bad:

```js
user.name = "Rahul";
setUser(user);
```

✅ Good:

```js
setUser({
  ...user,
  name: "Rahul",
});
```

Because React cares about:

# new reference = new state

---

# 🧠 Immutability in React

React performance relies heavily on:

# immutable updates

Meaning:

- avoid mutating existing objects
- create new references

---

# 🚨 Why Mutation Is Dangerous

Mutation causes:

- stale UI
- memo bugs
- caching bugs
- unpredictable rerenders

Especially in large apps.

---

# ⚡ Shallow Copy vs Deep Copy

When copying objects in JavaScript, there are TWO important concepts:

- shallow copy
- deep copy

Understanding this prevents massive bugs.

---

# 🟦 Shallow Copy

A shallow copy copies only the FIRST level.

Example:

```js id="lq2x0g"
const copy = { ...obj };
```

This creates a NEW outer object.

BUT nested objects inside are still shared.

---

# Example

```js id="t6k7vz"
const a = {
  name: "Aman",
  nested: {
    x: 1,
  },
};
```

Memory:

```txt id="n1g5pr"
a ──> 0x001

0x001
{
  name: "Aman",
  nested ──> 0x100
}

0x100
{
  x: 1
}
```

---

# Create Shallow Copy

```js id="cvw5q2"
const b = { ...a };
```

Now:

```txt id="t7m2kh"
b ──> 0x002
```

New outer object created.

BUT:

```txt id="7r3kfd"
b.nested ──> 0x100
```

Still SAME nested object.

---

# Important Check

```js id="md0s3w"
a === b;
```

Result:

```js id="gzx1ha"
false;
```

Different outer objects.

---

But:

```js id="z5h2ul"
a.nested === b.nested;
```

Result:

```js id="q9v4nt"
true;
```

Because BOTH point to same nested object in memory.

---

# Why This Is Dangerous

If you change nested data:

```js id="u3q4zi"
b.nested.x = 999;
```

Then:

```js id="z89g9j"
console.log(a.nested.x);
```

Output:

```js id="6gxut0"
999;
```

Even though you changed `b`.

Why?

Because:

```txt id="4yq98w"
a.nested and b.nested
```

are the SAME object.

---

# Easy Mental Model

## Shallow Copy

```txt id="ywl8x7"
new outer box
same inner boxes
```

---

# 🟥 Deep Copy

Deep copy creates completely independent objects.

No shared nested references.

---

# Example

```js id="uw5b8r"
const a = {
  nested: {
    x: 1,
  },
};

const b = structuredClone(a);
```

Now:

```js id="12x6tf"
a.nested === b.nested;
```

Result:

```js id="vowj4u"
false;
```

Different nested objects.

---

# Memory View

```txt id="1w8q6v"
a ──> 0x001
nested ──> 0x100

b ──> 0x002
nested ──> 0x200
```

Completely separate memory.

---

# Now Mutation Is Safe

```js id="0b6ft0"
b.nested.x = 999;
```

Does NOT affect:

```js id="r7n3qy"
a.nested.x;
```

because references are different.

---

# React Insight

This matters in React because shallow copying alone is NOT enough for deeply nested state.

Bad:

```js id="q5qpsk"
const newUser = { ...user };

newUser.address.city = "Delhi";
```

You still mutated nested state.

Because:

```txt id="jlwm1v"
user.address === newUser.address
```

is still true.

---

# Correct Nested Update

```js id="13m38i"
setUser({
  ...user,
  address: {
    ...user.address,
    city: "Delhi",
  },
});
```

Now every changed level gets a NEW reference.

React can detect updates correctly.
