Consistency = "Rules Never Break" - Database stays valid before + after transaction.

```sql
Table: Accounts (A, B)  -- Check constraint: A + B = 100 ALWAYS

Start: A=60, B=40  ✅ (60+40=100)

BEGIN TRANSACTION;
UPDATE Accounts SET A = 70 WHERE id=1;  -- Now A=70, B=40 ❌ (110!)
-- FAIL! Constraint violation → ROLLBACK → Back to 60+40

Only this works:
UPDATE Accounts SET A = 70, B = 30;  -- 70+30=100 ✅ COMMIT

```

### What gets checked

```text
✅ Primary Key UNIQUE
✅ Foreign Key EXISTS
✅ NOT NULL filled
✅ Check constraints (A + B = 100)
✅ Data types (no 'abc' in INT)
✅ Custom business rules
```

```text
Bank: No negative balances
Hospital: Age > 0
Store: Price > 0
Inventory: stock >= 0

Transaction EITHER keeps ALL rules valid
OR gets REJECTED entirely
```

### Example

```sql
CREATE TABLE account_balances (
  pair_id INT,
  alice_balance DECIMAL,
  bob_balance DECIMAL,
  CHECK (alice_balance + bob_balance = 100),  -- HERE!
  PRIMARY KEY (pair_id)
);

-- Insert initial state:
INSERT INTO account_balances (pair_id, alice_balance, bob_balance)
VALUES (1, 60, 40);  -- 60+40=100 ✅

-- This FAILS:
BEGIN;
UPDATE account_balances SET alice_balance = 70 WHERE pair_id = 1;
-- 70+40=110 ≠ 100 → CHECK FAIL → ROLLBACK ❌

-- This SUCCEEDS:
BEGIN;
UPDATE account_balances SET alice_balance = 70, bob_balance = 30 WHERE pair_id = 1;
-- 70+30=100 ✅ COMMIT

-- Most Common: Simple CHECK constraints
CHECK (balance >= 0)           -- No negatives
CHECK (price > 0)              -- Valid prices
CHECK (stock >= ordered_qty)   -- Can't oversell

```
